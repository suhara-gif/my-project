# チェックリスト型スキルは実行スクリプトへ寄せ、negative test で「壊れたら FAIL するか」を検証する

- 日付: 2026-07-14
- 種別: 設計判断
- 触れたファイル: .claude/skills/shell-safety-check/SKILL.md, .claude/skills/shell-safety-check/scripts/check.sh:1

## 問題
Anthropic の Skill 作成ガイドを踏まえて既存スキルを見直したところ、`shell-safety-check`
が「9 個の bash ブロックをモデルが毎回手で写経する」構成になっていた。写経ミス・省略が
起きうるうえ、CI から同じ検証を再利用できない。

## 文脈
ガイドの中心原則の一つが「決定的な処理はモデルに再現させず実行スクリプトに寄せる
(progressive disclosure: SKILL.md は薄い入口、詳細・実行体は同梱リソース)」。
これは本リポジトリ自身の設計核(①固め=機械的・決定的)と完全に一致する。
一方で、単にスクリプト化しただけでは「常に緑を返すだけのラバースタンプ」に退化する危険がある。

## 解法と、その理由
- 検証本体を `scripts/check.sh` に集約し、SKILL.md は「これを走らせて出力を読む」+
  何を守るかの表だけに薄くした(段階開示)。CI からも同一スクリプトを呼べる。
- チェッカーは複数チェックを最後まで走らせて集計するため `set -e` を**意図的に外し**、
  `set -uo pipefail` + fail カウンタで実装(理由をスクリプト冒頭に明記)。ゲート型スクリプトの
  正当な例外。
- ハード不変条件(秘密除外の欠落・再帰ガード欠落・権限逸脱・プロンプト非テンプレ化)は
  「トークンの存在アサート」にして、欠けたら exit 1。目視だった 4〜8 を自動 FAIL に変えた。
- **最重要**: 実装後に「使い捨てコピーで `*.key` 除外を1行消す」negative test を行い、
  ゲートが確かに FAIL するのを確認した。これをやらないと「守っているつもり」で通る。
- ルート移動は `git rev-parse --show-toplevel`(無ければ dirname 4段上)で BSD/GNU 両対応。
  `realpath`/`readlink -f` は使わない。

## うまくいかなかったこと
- 当初 `set -euo pipefail` を付けようとしたが、`grep` の no-match(exit 1)や失敗チェックで
  スクリプト自体が途中終了し、集計ゲートとして成立しない。ゲートでは `-e` を外すのが正しい。

## 抽出したルール / ヒューリスティック
- 決定的なチェックリスト型スキルは、モデルに写経させず `scripts/*.sh` に寄せ、SKILL.md は
  入口に薄くする(progressive disclosure)。
- 検証ゲートを作ったら、必ず不変条件を1つ壊した使い捨てコピーで「FAIL するか」を確認する。
  FAIL を再現できないゲートはラバースタンプであり、無いより危険。
- 複数チェックを集計するゲートは `set -e` を外し(`set -uo pipefail` + fail カウンタ)、
  外した理由をスクリプト冒頭に明記する。
