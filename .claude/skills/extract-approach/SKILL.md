# extract-approach — 学びノート記録スキル

## 目的

モデルが問題を解くと、その「なぜそう解いたか」はセッション終了で消える。このスキルは
その推論を**リポジトリ内の読めるノート**として固定する。解けた答えそのものより、
次に同種の問題へ当たったときに効く**判断基準**を残すのが狙い。

## いつ実行するか

CLAUDE.md の learning law に従い、**非自明な問題を1つ解き終えた直後、次の作業に移る前**に実行する。
対象の例: 厄介なバグの原因特定、アーキテクチャ上の選択、移植性(BSD/GNU)の罠、
何かを試して**うまくいかなかった**こと(失敗も同じ価値がある)。

自明な変更(タイポ修正、コメント微修正)には実行しない。ノートを量産して薄めない。

## 手順

1. `docs/learnings/` が無ければ作る。
2. ファイル名は `docs/learnings/<YYYYMMDD>-<短いスラッグ>.md`。
   日付は今日の日付、スラッグは英小文字とハイフンで問題を一言表す(例 `20260711-bsd-date-portability.md`)。
   同名があれば末尾に `-2` 等を付ける。
3. 下記テンプレートで**1問1ノート**を書く。長い総合レポートにしない — アトミックに保つ。
4. 事実ベースで書く。憶測や誇張を混ぜない。うまくいかなかったことは正直に書く。
5. 関連する既存ノートがあれば `関連` 欄に相対リンクで繋ぐ(Obsidian 風の相互リンク)。

## テンプレート

```markdown
# <一行タイトル: 何を解いたか>

- 日付: <YYYY-MM-DD>
- 種別: バグ修正 / 設計判断 / 移植性 / 調査 / 失敗した試み
- 触れたファイル: <path:line ...>

## 問題
<何が起きていたか。症状・エラー・意思決定が必要だった点を具体的に。>

## 文脈
<なぜ非自明だったか。制約(例: macOS bash 3.2 対応、データを失えない 等)。>

## 解法と、その理由
<採った手順と、なぜそれを選んだか。他候補を退けた理由も。ここが本体。>

## うまくいかなかったこと
<試して外した方法と、外した理由。無ければ「なし」。>

## 抽出したルール / ヒューリスティック
<次に同種の問題へ当たったとき能力の低いモデルでも従える、チェック可能な一文。
 一般性があれば CLAUDE.md の「壊しやすい間違い」への追記も検討する。>

## 関連
<関連する既存ノートへの相対リンク。無ければ省略。>
```

## 完了条件

- [ ] `docs/learnings/` に新しいノートが1件ある。
- [ ] 「解法と理由」と「抽出したルール」が埋まっている(空欄で出さない)。
- [ ] ルールがチェック可能な文になっている(形容詞の羅列ではない)。
- [ ] 抽出したルールが一般的なら CLAUDE.md への反映を検討した。
